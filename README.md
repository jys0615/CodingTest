## 파일명 규칙

| 접미사 | 의미 |
|---|---|
| `1260.py` | 정상 풀이 |
| `1260_err.py` | 틀렸거나 한 번에 못 푼 풀이 |
| `1260_fail.py` | 접근 자체가 틀린 풀이 (논리 오류) |
| `1260_again.py` | 복습 차원에서 다시 푼 풀이 |
| `1260_study.py` | 개념 학습용으로 따라 짠 코드 |
| `1260_review.py` | 정답 본 후 창 닫고 다시 직접 짠 코드 |

**심화 팁**: `_err` 뒤에 이유를 붙이면 자신의 약점 패턴이 보인다.

```
1260_err_오버플로우.py      # 큰 수 처리 실수
1260_err_엣지케이스.py      # 경계값 처리 누락
1260_err_로직.py            # 알고리즘 설계 오류
1260_err_구현.py            # 알고리즘은 맞는데 코드 실수
```

## 문제를 풀었을 때

1. **더 나은 풀이가 있는가?** 시간복잡도와 코드 간결함을 스스로 검토한다.
2. **이 패턴을 언제 또 쓸 수 있는가?** 한 줄만 메모한다.
   - 예: *"구간 합 → prefix[r+1] - prefix[l] 공식, 1-indexed로 만들 것"*
3. 풀고 넘어갈 때가 아는 것과 설명할 수 있는 것의 차이가 가장 큰 순간이다. 바로 넘어가지 않는다.

## 문제를 틀리거나 못 풀었을 때

### 단계별 프로세스

```
1. 20~30분 고민 → 막히는 지점 파악
         ↓
2. 정답 코드 X, 힌트만 탐색 (어떤 알고리즘인지 확인)
         ↓
3. 힌트 보고 다시 시도
         ↓
4. 그래도 안 되면 정답 코드 확인 → 완전히 이해
         ↓
5. 창 닫고, 처음부터 직접 다시 짜기  ← 이 단계가 핵심
         ↓
6. _err 또는 _fail 파일로 저장, 이유 기록
         ↓
7. 3일 후 다시 풀기 (_again)
```

### 왜 "창 닫고 다시 짜기"인가

코테 현장에서 막히는 이유는 거의 항상 **"본 적은 있는데 왜 그렇게 하는지 몰랐던"** 문제다. 이해한 것과 직접 짤 수 있는 것은 완전히 다르다.

## 복습 주기

| 상태 | 다음 복습 |
|---|---|
| 처음 틀린 문제 (`_err`, `_fail`) | 3일 후 |
| `_again` 풀었는데 또 틀림 | 1주일 후 다시 |
| `_again` 풀어서 성공 | 접미사 제거, 마무리 |
| 여러 번 틀린 문제 (예: `_err6`) | 취약 문제 목록에 별도 기록 |

## 유형별 공부 팁

**입출력/기초** — 실수(float) 입력, 빠른 입력(`sys.stdin`) 습관화

**정렬** — `key` 함수 다중 조건 정렬 (`lambda x: (x[1], x[0])`) 손에 익히기

**자료구조** — 스택/큐/힙 각각 언제 쓰는지 판단 기준 정리

**그리디** — 증명이 핵심. "왜 욕심내도 되는가"를 말로 설명할 수 있어야 함

**DFS/BFS** — 재귀 DFS는 깊은 그래프에서 스택 오버플로우 주의 (`sys.setrecursionlimit`)

**백트래킹** — 가지치기 조건이 성능을 결정함. 조건을 최대한 앞으로 당기기

**이분탐색** — `lo <= hi` vs `lo < hi` 구분, 매개변수 탐색 패턴 암기

**DP** — 점화식을 말로 먼저 정의, 그 다음 코드로 옮기기

**최단경로** — 다익스트라는 음수 간선 불가, 플로이드는 전체 쌍 최단거리

## _암기_노트 활용법

`_암기_노트/` 폴더의 파일들은 주석만 보고 코드를 직접 타이핑하는 방식으로 사용한다.

```
1. 파일 열기
2. 주석(설명)만 보고 코드 가림
3. 직접 타이핑
4. 막힌 부분만 확인
5. 반복
```

코테 직전에는 이 파일들만 순서대로 타이핑해도 패턴이 머릿속에 정착된다.
